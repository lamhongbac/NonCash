# NonCash POS Integration Guide

This guide is for third-party POS vendors, payment integrators, and mall IT teams who want to enable voucher redemption at physical or digital checkout counters through the NonCash platform.

---

## 1. Overview

NonCash exposes a set of stateless, atomic POS endpoints under `/api/v1/pos`. Each outlet is identified by an API key. The recommended redemption flow is:

```
Verify → Lock → (payment completes) → Commit
           ↓
     (payment fails) → Rollback
```

- **Verify** — Check that a voucher is valid for the current outlet and bill.
- **Lock** — Atomically reserve the voucher for 10 minutes (state: `InUse`).
- **Commit** — Finalize redemption and record a `VoucherUsage`.
- **Rollback** — Release the lock if the transaction is cancelled.

All endpoints are idempotent where noted, making them safe to retry on network failures.

---

## 2. Authentication

Every request to `/api/v1/pos/*` must include the outlet-specific API key.

| Header | Value | Example |
|---|---|---|
| `X-API-Key` | The outlet's `ApiKeyPrefix` | `OUTLET_KEY_ABC123` |

The API key is resolved to an outlet by `ApiKeyMiddleware`. The `OutletId` supplied in the request body must match the outlet attached to the API key; otherwise the request is rejected with `OutletNotAuthorized`.

> **Security note:** Treat API keys as secrets. In production they should be rotated and stored securely on the POS device or backend, never hard-coded in client-side code.

---

## 3. Voucher code format

A voucher code is a cryptographically signed string generated by NonCash. The POS does not need to understand the internal structure; it simply passes the full code to the API.

Example: `NCF-7A2B9C4D-8F3E1A6B`

## 4. API endpoints

### 4.1 Verify a voucher

Check whether a voucher is valid at a specific outlet. This call is read-only and does **not** change voucher state.

```http
POST /api/v1/pos/verify
X-API-Key: {outlet_api_key}
Content-Type: application/json

{
  "voucherCode": "NCF-7A2B9C4D-8F3E1A6B",
  "outletId": "11111111-1111-1111-1111-111111111111"
}
```

**Response — Valid:**

```json
{
  "status": "Valid",
  "reason": null,
  "voucherInfo": {
    "faceValue": 100000,
    "valueType": "FixedAmount",
    "expiryDate": "2026-12-31T23:59:59Z",
    "brandName": "Giga Coffee",
    "serialNo": "VC-2026-00001"
  }
}
```

**Response — Invalid:**

```json
{
  "status": "Invalid",
  "reason": "Expired",
  "voucherInfo": null
}
```

Common reasons: `Forged`, `AlreadyUsed`, `Expired`, `NotYetValid`, `OutletNotAuthorized`, `TransferPending`.

---

### 4.2 Lock a voucher

Reserve the voucher for a specific bill. The voucher state changes to `InUse` and a `LockId` is returned. The lock expires after **10 minutes** if not committed or rolled back.

```http
POST /api/v1/pos/lock
X-API-Key: {outlet_api_key}
Content-Type: application/json

{
  "voucherCode": "NCF-7A2B9C4D-8F3E1A6B",
  "outletId": "11111111-1111-1111-1111-111111111111",
  "billNumber": "BILL-2026-000123"
}
```

**Response:**

```json
{
  "status": "Locked",
  "reason": null,
  "lockId": "22222222-2222-2222-2222-222222222222",
  "voucherInfo": { ... }
}
```

**Idempotency:** Calling `lock` again with the same `(voucherCode, outletId, billNumber)` returns the existing `LockId` and HTTP 200.

**Conflicts:** If the voucher is already locked for a different bill or outlet, the API returns HTTP 409 with `status: "AlreadyInUse"`.

---

### 4.3 Commit a voucher

Finalize redemption. This call records a `VoucherUsage`, marks the voucher as `Complete`, and removes the lock.

```http
POST /api/v1/pos/commit
X-API-Key: {outlet_api_key}
Content-Type: application/json

{
  "lockId": "22222222-2222-2222-2222-222222222222",
  "transactionId": "TXN-2026-000123-001",
  "amountUsed": 100000
}
```

**Response:**

```json
{
  "status": "Success",
  "reason": null,
  "message": "Voucher completed"
}
```

**Idempotency:** Repeating the same `transactionId` returns HTTP 200 with `status: "Success"` and a message indicating the voucher was already committed.

**Expired lock:** If the lock has expired, the API returns HTTP 409 with `status: "LockExpired"`. The POS should re-call `lock` before committing.

---

### 4.4 Rollback a voucher

Release a lock without recording usage. Use this when the customer cancels the payment or the voucher is no longer needed.

```http
POST /api/v1/pos/rollback
X-API-Key: {outlet_api_key}
Content-Type: application/json

{
  "lockId": "22222222-2222-2222-2222-222222222222"
}
```

**Response:**

```json
{
  "status": "Success",
  "reason": null,
  "message": "Voucher released"
}
```

**Idempotency:** If the lock is already released, the API returns HTTP 200 with `status: "AlreadyReleased"`. If the voucher was already committed, it returns HTTP 409 with `status: "AlreadyCompleted"`.

## 5. Recommended POS integration patterns

### Pattern A — Native POS integration

The POS software calls NonCash POS endpoints directly during checkout.

1. Cashier scans voucher code.
2. POS calls `/verify` and displays voucher value.
3. POS calls `/lock` when the customer confirms payment.
4. POS completes its own payment flow.
5. POS calls `/commit` on success or `/rollback` on cancellation.

Best for: modern POS systems that support HTTP/REST and can be customized.

### Pattern B — Standalone redemption terminal

For POS systems that cannot be customized, use a separate device (tablet, phone, or browser) running the NonCash POS client. The cashier:

1. Opens the NonCash POS client (`/pos/redeem`).
2. Selects the outlet.
3. Scans or types the voucher code.
4. Locks and commits the voucher.
5. Applies the discount manually on the main POS.

Best for: legacy POS systems, small tenants, or temporary pop-up stores.

### Pattern C — Middleware / payment gateway

A mall-level middleware receives voucher codes from multiple POS systems, calls NonCash on their behalf, and returns results in a unified format.

Best for: large malls with many different POS vendors and a central IT team.

## 6. Error handling

All POS endpoints return HTTP 200 for business-as-usual outcomes, even when the voucher is invalid. Check the `status` field in the response body.

| HTTP status | Meaning | Recommended action |
|---|---|---|
| 200 OK | Request processed; check `status` field | Inspect `status` and `reason` |
| 401 Unauthorized | Missing or invalid `X-API-Key` | Check API key configuration |
| 409 Conflict | Lock conflict or already committed | Display appropriate message to cashier |
| 400 BadRequest | Malformed request | Fix request payload |
| 500 Internal Server Error | Server error | Retry with backoff, contact support |

### Common `status` / `reason` values

| Status | Reason | Meaning |
|---|---|---|
| `Valid` | — | Voucher can be used |
| `Invalid` | `Forged` | Code is not a valid NonCash voucher |
| `Invalid` | `Expired` | Voucher past expiry date |
| `Invalid` | `NotYetValid` | Voucher not yet active |
| `Invalid` | `AlreadyUsed` | Voucher already redeemed |
| `Invalid` | `OutletNotAuthorized` | Voucher not valid at this outlet |
| `Invalid` | `TransferPending` | Voucher is locked in a pending peer-to-peer transfer |
| `Locked` | — | Lock acquired |
| `AlreadyInUse` | — | Voucher locked by another transaction |
| `LockExpired` | — | Lock timed out; re-lock before commit |
| `AlreadyComplete` | — | Voucher already committed |
| `AlreadyReleased` | — | Lock already rolled back |

## 7. Idempotency and retry safety

| Endpoint | Idempotency key | Behavior on retry |
|---|---|---|
| `/verify` | Stateless | Safe to call repeatedly |
| `/lock` | `(voucherCode, outletId, billNumber)` | Returns same `LockId` if already locked for same bill |
| `/commit` | `transactionId` | Returns success if same transactionId was already committed |
| `/rollback` | `lockId` | Returns `AlreadyReleased` if already rolled back |

POS implementations should generate a stable `transactionId` (e.g., `POS_ID + billNumber + sequential counter`) and retry `/commit` on transient failures.

---

## 8. Security checklist

- [ ] API keys are stored securely on the POS or middleware, not in source code.
- [ ] API keys are rotated periodically.
- [ ] POS devices use HTTPS when calling NonCash.
- [ ] `transactionId` is unique and non-predictable.
- [ ] Lock/commit timeouts are handled gracefully to avoid double redemption.
- [ ] POS UI shows clear success/failure messages to the cashier.

---

## 9. Testing checklist

Before going live, validate:

- [ ] Valid voucher → verify returns `Valid`.
- [ ] Invalid/forged code → verify returns `Invalid`/`Forged`.
- [ ] Wrong outlet → verify returns `OutletNotAuthorized`.
- [ ] Lock → voucher cannot be locked again for a different bill.
- [ ] Commit → voucher cannot be reused.
- [ ] Rollback → voucher can be locked again.
- [ ] Expired lock → commit returns `LockExpired`.
- [ ] Duplicate transactionId → commit returns success without double usage.
